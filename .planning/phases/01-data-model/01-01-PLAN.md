  <action>
Create migration `005_vlan_trunk_model.sql` that performs the following steps IN ORDER:

**Step 1: Add new columns to interfaces table**
SQLite cannot DROP columns (pre-3.35) so use the rename-create-copy-drop pattern:
1. `ALTER TABLE interfaces RENAME TO interfaces_old;`
2. Create new `interfaces` table with same columns EXCEPT: remove `role` and `vlan_id`, add `pvid INTEGER NOT NULL DEFAULT 10` and `tagged_vlans TEXT NOT NULL DEFAULT '[]'`. Keep all other columns: id, name, mac, ips, mtu, is_up, enabled, config. Keep the UNIQUE constraint on name, AUTOINCREMENT on id.
3. Copy data with role-to-pvid mapping:
   ```sql
   INSERT INTO interfaces (id, name, mac, ips, mtu, is_up, pvid, tagged_vlans, enabled, config)
   SELECT id, name, mac, ips, mtu, is_up,
     CASE
       WHEN role = 'lan' THEN 10
       WHEN role = 'mgmt' THEN 3000
       WHEN role = 'guest' THEN 3001
       WHEN role = 'dmz' THEN 3002
       WHEN role = 'wan' THEN 0
       ELSE 10
     END,
     '[]',
     enabled, config
   FROM interfaces_old;
   ```
   WAN ports get pvid=0 because WAN is a completely separate world from internal VLANs — WAN PVID is meaningless in the internal VLAN scheme. The value 0 signals "not an internal VLAN port".
4. `DROP TABLE interfaces_old;`

**Step 2: Insert VLAN 1 void entry into networks**
```sql
INSERT OR IGNORE INTO networks (name, zone, vlan_id, subnet, gateway, dhcp_enabled, enabled)
VALUES ('Void', 'void', 1, '0.0.0.0/32', '0.0.0.0', 0, 0);
```
This creates the void VLAN 1 entry — it exists in the DB so the system knows about it, but enabled=0 and it gets no bridge, no DHCP, no subnet. Any traffic on VLAN 1 will be DROPped by firewall rules (Phase 3).

**Step 3: Update LAN network to VLAN 10**
```sql
UPDATE networks SET vlan_id = 10 WHERE zone = 'lan' AND (vlan_id IS NULL OR vlan_id = 1);
```
This changes LAN from untagged/VLAN 1 to VLAN 10. Only updates if currently NULL or 1 (doesn't override if user already changed it).

Add a comment header explaining the migration purpose and the role→PVID mapping rationale.

IMPORTANT: Use parameterized-safe SQL only (no dynamic values, this is a static migration). All values are literals.
