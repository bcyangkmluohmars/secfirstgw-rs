<task type="auto">
  <name>Task 1: Rewrite load_interface_zones() to use PVID + networks JOIN</name>
  <files>crates/sfgw-fw/src/lib.rs</files>
  <action>
The current `load_interface_zones()` queries `SELECT name, role FROM interfaces WHERE enabled = 1` — the `role` column was removed in migration 005. This crashes at runtime. Rewrite it to derive zones from PVID via the networks table.

**New SQL strategy:**

1. For internal interfaces (pvid > 0): JOIN interfaces to networks on `interfaces.pvid = networks.vlan_id` to get the zone name. Query:
   ```sql
   SELECT i.name, n.zone FROM interfaces i
   JOIN networks n ON i.pvid = n.vlan_id
   WHERE i.enabled = 1 AND i.pvid > 0
   ```

2. For WAN interfaces (pvid = 0): Select them separately as WAN zone:
   ```sql
   SELECT name FROM interfaces WHERE enabled = 1 AND pvid = 0
   ```

3. Build `zone_map: HashMap<String, Vec<String>>` the same way as before — zone name to list of interface names.

4. The bridged zone logic (replacing individual interfaces with `br-{zone}`) stays the same — query `networks` for enabled zones to determine which are bridged.

5. WAN interfaces keep their raw names (eth8, eth9) — they are not bridged.

**Key points:**
- `from_role()` can be renamed to `from_zone_name()` or kept as-is since it's just string→enum parsing. Either way update the doc comment to not reference the removed `role` column.
- Update the doc comment on `load_interface_zones()` to describe the PVID-based derivation.
- Interfaces with pvid values that don't match any network's vlan_id are silently excluded (they have no zone assignment yet — user must configure).
- The `FirewallZone::from_role` function name is fine to keep (it's internal), but update the doc comment from "Parse a zone from the `role` column" to "Parse a zone from a zone name string".
  </action>

<task type="auto">
  <name>Task 2: Add VLAN isolation rules to iptables generation</name>
  <files>crates/sfgw-fw/src/iptables/mod.rs, crates/sfgw-fw/src/iptables/tests.rs</files>
  <action>
Add VLAN isolation rules to `generate_zone_ruleset()` that enforce WAN-01, WAN-02, WAN-03, FW-01, FW-02. These rules go BEFORE the existing zone rules (right after `emit_default_rules(out)`), since VLAN isolation must be checked first.

**New function: `emit_vlan_isolation_rules(out, wan_ifaces, internal_ifaces)`**

This function emits iptables rules in the SFGW-INPUT and SFGW-FORWARD chains:

1. **FW-02: VLAN 1 void catch-all DROP** — Drop any packet on any VLAN 1 sub-interface. Since VLAN 1 has no bridge (Phase 2 decision), traffic would only appear on VLAN sub-interfaces like `eth0.1`. Add rules:
   ```
   -A SFGW-INPUT -i +.1 -j DROP -m comment --comment "VLAN 1 void DROP"
   -A SFGW-FORWARD -i +.1 -j DROP -m comment --comment "VLAN 1 void DROP"
   ```
   Note: iptables `+` is a wildcard, so `+.1` matches any interface ending in `.1`. However, this is NOT correct iptables syntax. Instead, use a more targeted approach: iterate over known physical interfaces (from zones) and add `{iface}.1` DROP rules. OR, since the bridge for VLAN 1 doesn't exist (Phase 2), we can use a comment and rely on the fact that no bridge exists. BUT for defense-in-depth, add explicit rules.

   **Better approach:** The VLAN 1 isolation is best done by matching on the 802.1Q tag using the `physdev` module is not reliable. Instead, since no bridge is created for VLAN 1, traffic can only arrive on raw VLAN sub-interfaces. But those sub-interfaces should not exist either. The real defense-in-depth is: if someone creates a VLAN 1 sub-interface (e.g., `eth0.1`), DROP all traffic on it.

   **Simplest correct approach:** Add rules that match any interface name matching the pattern `*.1` (VLAN 1 sub-interfaces). In iptables, the `-i` flag supports `+` as suffix wildcard but NOT infix wildcards. So we cannot match `*.1` generically. Instead:
   - Get all known physical base interfaces (from all zones — eth0 through eth9, etc.)
   - For each base interface, add: `-A SFGW-INPUT -i {base}.1 -j DROP`
   - For each base interface, add: `-A SFGW-FORWARD -i {base}.1 -j DROP`

   **Even simpler:** Since sfgw-fw knows the zone interfaces, extract the base physical interface names from the zone policies. For WAN interfaces like `eth8`, `eth9`, add `eth8.1`, `eth9.1` DROP rules. For bridged interfaces, traffic comes through the bridge — but since there's no `br-void` bridge, VLAN 1 frames on bridged ports are already dropped at the switch ASIC level (Phase 2). Still add DROP rules for defense-in-depth on any sub-interface named `*.1`.

   **Final approach:** Accept a list of all known base interface names. For each, add VLAN 1 DROP rules. Pass these as a parameter to the function. The base interfaces can be collected from the ZonePolicy list by extracting raw interface names (before bridging) from a new helper, OR from a separate DB query. For simplicity, pass the WAN interface names (they're the unbridged ones where a `.1` sub-interface could theoretically exist) and emit DROP rules for `{wan}.1`. For bridged interfaces, the bridge handles it — but add a catch-all: if somehow `br-void` or `br-void.1` appears, DROP it too.

   **Pragmatic implementation:**
   - For each WAN interface: `-A SFGW-INPUT -i {wan}.1 -j DROP` and `-A SFGW-FORWARD -i {wan}.1 -j DROP`
   - Also add a paranoid rule: `-A SFGW-INPUT -i br-void -j DROP` and `-A SFGW-FORWARD -i br-void -j DROP` (in case someone creates it)

2. **WAN-01: Internal VLANs blocked on WAN interfaces** — For each WAN interface, DROP packets tagged with internal VLAN IDs. Internal VLANs are: 10 (LAN), 3000 (MGMT), 3001 (Guest), 3002 (DMZ). These would arrive as sub-interfaces `{wan}.10`, `{wan}.3000`, etc. Add rules:
   ```
   -A SFGW-INPUT -i {wan}.10 -j DROP -m comment --comment "no internal VLAN on WAN"
   -A SFGW-INPUT -i {wan}.3000 -j DROP -m comment --comment "no internal VLAN on WAN"
   ... (same for 3001, 3002, and same for SFGW-FORWARD)
   ```

   **Better approach:** Instead of hardcoding VLAN IDs, derive them from the `zones` parameter. For each zone that is NOT WAN and NOT void, get its VLAN ID from the interface name (e.g., `br-lan` → the zone is `lan` → look up its vlan_id). But we don't have vlan_id in ZonePolicy currently.

   **Simplest correct approach:** Add `vlan_id: Option<u16>` to `ZonePolicy` struct. Populate it from the networks table in `load_interface_zones()`. Then in `emit_vlan_isolation_rules`, for each non-WAN zone with a vlan_id, add DROP rules on WAN interfaces for that VLAN ID.

   Update `ZonePolicy` to include:
   ```rust
   pub struct ZonePolicy {
       pub zone: FirewallZone,
       pub interfaces: Vec<String>,
       pub vlan_id: Option<u16>,  // NEW: VLAN ID from networks table
   }
   ```

   In `load_interface_zones()`, populate `vlan_id` from the networks table query.

   Then in `emit_vlan_isolation_rules`:
   ```
   for zone in zones where zone.zone != WAN and zone.vlan_id.is_some():
       let vid = zone.vlan_id.unwrap()
       for wan_iface in wan_ifaces:
           -A SFGW-INPUT -i {wan_iface}.{vid} -j DROP
           -A SFGW-FORWARD -i {wan_iface}.{vid} -j DROP
   ```

3. **WAN-02 + WAN-03: Provider VLANs blocked on internal interfaces** — Provider VLANs are configured per-WAN interface in the `config` JSON column (e.g., `{"vlan": 7}` for Telekom). For now, we don't have provider VLAN config in the DB yet (that's a WAN config detail). The enforcement here is simpler: any traffic arriving on an internal (non-WAN) interface that is tagged with a VLAN ID NOT in the internal VLAN set should be dropped. But iptables can't inspect 802.1Q tags directly without `ebtables`.

   **Practical approach for WAN-02/WAN-03:** The switch ASIC (Phase 2) already enforces VLAN membership at the hardware level — a LAN port configured with pvid=10 will only accept VLAN 10 frames. The firewall rules are defense-in-depth. The key rule is: internal bridge interfaces (`br-lan`, `br-mgmt`, etc.) should NOT see any traffic from WAN-associated VLANs. Since WAN interfaces are not bridged and have pvid=0, they never join any internal bridge. The separation is inherent.

   **What we CAN do as defense-in-depth:** If a WAN interface has a provider VLAN sub-interface (e.g., `eth8.7` for Telekom VLAN 7), ensure that sub-interface never gets forwarded to internal zones. Since WAN → any is already DROP in the forward chain (existing rule at priority 400), this is already handled. BUT: add explicit DROP rules for WAN VLAN sub-interfaces in INPUT too, for completeness.

   For WAN-02/WAN-03, the existing zone rules already handle this (WAN → any = DROP forward, WAN input = DROP). The VLAN isolation rules above (WAN-01) prevent internal VLANs from appearing on WAN. The reverse (provider VLANs on internal) is prevented by the switch ASIC. No additional iptables rules needed beyond what's already present + the WAN-01 rules above.

4. **FW-01: LAN zone references VLAN 10** — This is already handled by `load_interface_zones()` rewrite in Task 1. The LAN zone will resolve to `br-lan` (bridge for zone=lan, vlan_id=10). Verify in tests that the generated rules use `br-lan` not `br-1` or any VLAN 1 reference.

**Call site update in `generate_zone_ruleset()`:**

After `emit_default_rules(out)` and before the zone-specific rules, add:

