<task type="auto">
  <name>Task 1: Rewrite load_interface_zones() to use PVID + networks JOIN, add vlan_id to ZonePolicy</name>
  <files>crates/sfgw-fw/src/lib.rs</files>
  <action>
The current `load_interface_zones()` queries `SELECT name, role FROM interfaces WHERE enabled = 1` — the `role` column was removed in migration 005. This crashes at runtime. Rewrite it to derive zones from PVID via the networks table, and add `vlan_id` to `ZonePolicy` so Task 2 can derive VLAN isolation rules dynamically.

**ZonePolicy struct change:**

Add `vlan_id: Option<u16>` to `ZonePolicy`:
```rust
pub struct ZonePolicy {
    pub zone: FirewallZone,
    pub interfaces: Vec<String>,
    pub vlan_id: Option<u16>,  // NEW: VLAN ID from networks table (None for WAN)
}
```

Update all existing `ZonePolicy` construction sites to include `vlan_id`. WAN zones get `vlan_id: None`. Internal zones get `vlan_id: Some(n.vlan_id)` from the networks table query.

**New SQL strategy for load_interface_zones():**

1. For internal interfaces (pvid > 0): JOIN interfaces to networks on `interfaces.pvid = networks.vlan_id` to get the zone name and VLAN ID. Query:
   ```sql
   SELECT i.name, n.zone, n.vlan_id FROM interfaces i
   JOIN networks n ON i.pvid = n.vlan_id
   WHERE i.enabled = 1 AND i.pvid > 0
   ```

2. For WAN interfaces (pvid = 0): Select them separately as WAN zone:
   ```sql
   SELECT name FROM interfaces WHERE enabled = 1 AND pvid = 0
   ```

3. Build `zone_map: HashMap<String, Vec<String>>` the same way as before — zone name to list of interface names. Also track vlan_id per zone.

4. The bridged zone logic (replacing individual interfaces with `br-{zone}`) stays the same — query `networks` for enabled zones to determine which are bridged.

5. WAN interfaces keep their raw names (eth8, eth9) — they are not bridged.

**Key points:**
- `from_role()` can be renamed to `from_zone_name()` or kept as-is since it's just string->enum parsing. Either way update the doc comment to not reference the removed `role` column.
- Update the doc comment on `load_interface_zones()` to describe the PVID-based derivation.
- Interfaces with pvid values that don't match any network's vlan_id are silently excluded (they have no zone assignment yet — user must configure).
- The `FirewallZone::from_role` function name is fine to keep (it's internal), but update the doc comment from "Parse a zone from the `role` column" to "Parse a zone from a zone name string".

  <name>Task 2: Add VLAN isolation rules to iptables generation</name>
  <files>crates/sfgw-fw/src/iptables/mod.rs, crates/sfgw-fw/src/iptables/tests.rs</files>

Add VLAN isolation rules to `generate_zone_ruleset()` that enforce WAN-01, FW-01, FW-02. These rules go BEFORE the existing zone rules (right after `emit_default_rules(out)`), since VLAN isolation must be checked first.

**New function: `emit_vlan_isolation_rules(out, zones)`**

This function emits iptables rules in the SFGW-INPUT and SFGW-FORWARD chains:

1. **FW-02: VLAN 1 void DROP on WAN interfaces** — For each WAN interface, add explicit DROP rules for VLAN 1 sub-interfaces. Also add a paranoid DROP for `br-void` in case someone creates it:
   ```
   -A SFGW-INPUT -i {wan}.1 -j DROP -m comment --comment "VLAN 1 void DROP"
   -A SFGW-FORWARD -i {wan}.1 -j DROP -m comment --comment "VLAN 1 void DROP"
   ```
   And:
   ```
   -A SFGW-INPUT -i br-void -j DROP -m comment --comment "VLAN 1 void DROP"
   -A SFGW-FORWARD -i br-void -j DROP -m comment --comment "VLAN 1 void DROP"
   ```

   Rationale: WAN interfaces are unbridged, so a `.1` sub-interface could theoretically exist. For bridged (internal) interfaces, VLAN 1 frames are already dropped at the switch ASIC level (Phase 2) because VLAN 1 has no bridge. The `br-void` rules are defense-in-depth.

2. **WAN-01: Internal VLANs blocked on WAN interfaces** — Use the `vlan_id` field from `ZonePolicy` (added by Task 1). For each non-WAN zone that has a `vlan_id`, add DROP rules on every WAN interface for that VLAN ID:
   ```
   for zone in zones where zone.zone != WAN and zone.vlan_id.is_some():
       let vid = zone.vlan_id.unwrap()
       for wan_iface in wan_ifaces:
           -A SFGW-INPUT -i {wan_iface}.{vid} -j DROP -m comment --comment "no internal VLAN on WAN"
           -A SFGW-FORWARD -i {wan_iface}.{vid} -j DROP -m comment --comment "no internal VLAN on WAN"
   ```

   Extract `wan_ifaces` from zones: collect interfaces from zones where `zone.zone == WAN`.

3. **FW-01: LAN zone references VLAN 10** — This is already handled by `load_interface_zones()` rewrite in Task 1. The LAN zone will resolve to `br-lan` (bridge for zone=lan, vlan_id=10). Verify in tests that the generated rules use `br-lan` not `br-1` or any VLAN 1 reference.

**Note on WAN-02/WAN-03 (provider VLANs on internal interfaces):** These are enforced by the switch ASIC at the hardware level (Phase 2) — a LAN port with pvid=10 only accepts VLAN 10 frames. The existing zone rules (WAN -> any = DROP forward) provide additional defense-in-depth. No new iptables rules are needed for WAN-02/WAN-03.

**Call site update in `generate_zone_ruleset()`:**

After `emit_default_rules(out)` and before the zone-specific rules, add:
```rust
// VLAN isolation rules (before zone rules — must be checked first).
emit_vlan_isolation_rules(out, zones);
```
