Rewrite the switch VLAN programming to be driven by per-port config from the DB.

**1. Add DB query helper:**

Add `load_port_vlan_config(db) -> Result<Vec<PortVlanConfig>>` (async). Query:
```sql
SELECT name, pvid, tagged_vlans FROM interfaces WHERE pvid > 0
```
This returns only internal-VLAN ports (pvid > 0). WAN ports (pvid=0) are excluded -- they are outside the internal VLAN numbering space (decision from Phase 1).

Define a struct:
```rust
struct PortVlanConfig {
    name: String,
    pvid: u16,
    tagged_vlans: Vec<u16>,
}
```
Parse tagged_vlans from JSON string (same pattern as list_interfaces in lib.rs).

**2. Add port-name-to-switch-port mapping:**

Add a helper `iface_to_switch_port(name: &str, sw: &SwitchLayout) -> Option<u8>` that maps Linux interface names (eth0-eth7) to switch port numbers (0-7). On UDM Pro: eth0=port0, eth1=port1, ..., eth7=port7. This mapping follows the index in `sw.lan_ports` plus `sw.mgmt_port`. If the interface is not a switch-managed port, return None.

Note: The SwitchLayout knows port numbers but not interface names. The mapping is positional: lan_ports[0] = eth0, lan_ports[1] = eth1, etc. For MGMT port, it's always the port number in sw.mgmt_port. Build a lookup Vec or match on the name.

Actually, a simpler approach: build a mapping from ALL switch port numbers to interface names. On UDM Pro, switch ports 0-6 are eth0-eth6, port 7 is eth7 (MGMT). The mapping is `ethN = port N` for ports in lan_ports + mgmt_port. So the reverse is: strip "eth" prefix, parse the number, check if it's in lan_ports or is mgmt_port.

**3. Add `compute_vlan_port_map`:**

```rust
fn compute_vlan_port_map(
    ports: &[PortVlanConfig],
    sw: &SwitchLayout,
) -> BTreeMap<u16, Vec<PortMember>>
```

Where:
```rust
#[derive(Debug, Clone)]
struct PortMember {
    port: u8,
    tagged: bool,
}
```

Logic:
- For each PortVlanConfig, find its switch port number via iface_to_switch_port.
- The port's PVID VLAN gets the port as UNTAGGED.
- Each entry in tagged_vlans gets the port as TAGGED.
- After processing all ports, add CPU port (sw.cpu_port) as TAGGED to every VLAN in the map.
- Add internal ports (sw.internal_ports) as TAGGED to every VLAN in the map.
- Finally, add VLAN 1 entry: all lan_ports TAGGED + mgmt_port TAGGED + cpu_port TAGGED + internal_ports TAGGED. This is the void catch-all -- programmed on the ASIC so any unconfigured port's untagged traffic lands on VLAN 1, which has no bridge (dropped at CPU). Only add VLAN 1 if it's not already in the map (it shouldn't be since no port should have pvid=1 or tagged_vlans containing 1).

**4. Add `format_port_string`:**

Replace the old `build_port_string` with a simpler helper:
```rust
fn format_port_string(members: &[PortMember]) -> String
```
For each member: if tagged, "{port}t", else "{port}". Join with spaces.

**5. Rewrite `setup_switch_vlans`:**

Change signature to take `db: &sfgw_db::Db` instead of `networks: &[NetworkSetup]`.
- Call `load_port_vlan_config(db)` to get per-port config.
- Call `compute_vlan_port_map(&ports, sw)` to get per-VLAN port membership.
- Stale VLAN cleanup: iterate VLANs 1-4094 on the ASIC; if a VLAN has ports assigned but is NOT in our computed map, clear it. Use the existing `swconfig_vlan_has_ports` for the check (only check VLANs in the commonly-used range 1-100 plus any VLANs found as stale sub-interfaces, same approach as current code).
- For each VLAN in the map: call `swconfig_set_vlan_ports(device, vid, &format_port_string(&members))`.
- For each port with a known switch port number: call `swconfig_set_pvid(device, port, pvid)`. This replaces the MGMT-only PVID logic -- every port now gets its PVID set.
- Call `swconfig_apply` at the end.
- Keep void zone handling in `setup_bridges` unchanged (already skips void correctly).

**6. Update `setup_networks` caller:**

Since `setup_switch_vlans` now needs `db`, update the call in `setup_networks`:

- The function becomes async since it reads from DB.
- `load_enabled_networks` is still needed for `setup_bridges` (bridge setup is zone-driven, not port-driven).

**7. Remove old `build_port_string` and `NetworkSetup` zone-based VLAN logic:**

The old `build_port_string(vlan_id, zone, sw)` is no longer needed. Remove it. Keep `NetworkSetup` and `load_enabled_networks` because `setup_bridges` still uses them.
