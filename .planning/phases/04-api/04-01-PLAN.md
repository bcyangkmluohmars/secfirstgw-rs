**In `crates/sfgw-net/src/switch.rs`:**

Add a public `reconfigure_networks` function that re-runs network setup after a port config change. This wraps the existing `setup_networks` logic but is callable without a `SwitchLayout` (it detects the board at call time):

```rust
pub async fn reconfigure_networks(db: &sfgw_db::Db) -> Result<()>
```

Implementation:
- Call `sfgw_hal::detect_board()` to get optional switch layout (same pattern as `detect_interfaces_for_platform` in lib.rs)
- If board has switch info, build `SwitchLayout` and call `setup_networks(db, Some(&layout)).await`
- If no board (VM/Docker), call `setup_networks(db, None).await`
- This avoids duplicating the switch layout detection — it's the same 5-line pattern from `detect_interfaces_for_platform`

**In `crates/sfgw-api/src/lib.rs`:**

1. **Port GET endpoint** (`port_get_handler`):
   - Route: `GET /api/v1/ports/{name}`
   - Query: `SELECT name, mac, ips, mtu, is_up, pvid, tagged_vlans, enabled FROM interfaces WHERE name = ?1`
   - Return JSON: `{ name, mac, ips, mtu, is_up, pvid, tagged_vlans, enabled, speed, driver, port_type }` (reuse `detect_port_type`)
   - 404 if not found
   - No `role` field in response

2. **Port PUT endpoint** (`port_update_handler`):
   - Route: `PUT /api/v1/ports/{name}`
   - Accept JSON body with optional `pvid` (0 or 1-4094) and `tagged_vlans` (array of 1-4094)
   - Validate identically to existing `interface_update` (same pvid range check, same tagged_vlans element check)
   - After successful DB update:
     a. Drop the DB lock
     b. Call `sfgw_net::switch::reconfigure_networks(&db).await` to reprogram switch ASIC + bridges
     c. Call `sfgw_fw::apply_rules(&db).await` to regenerate firewall rules
     d. If either fails, log the error but still return 200 (DB is source of truth; reconfig will retry on next boot)
   - Return `{ "ok": true }` on success

3. **Zones list endpoint** (`zones_list_handler`):
   - Route: `GET /api/v1/zones`
   - Query: `SELECT id, name, zone, vlan_id, subnet, gateway, dhcp_enabled, enabled FROM networks ORDER BY zone`
   - Return JSON: `{ "zones": [...] }` where each zone has `{ id, name, zone, vlan_id, subnet, gateway, dhcp_enabled, enabled }`

4. **Zone GET endpoint** (`zone_get_handler`):
   - Route: `GET /api/v1/zones/{zone}`
   - Query: `SELECT id, name, zone, vlan_id, subnet, gateway, dhcp_enabled, enabled FROM networks WHERE zone = ?1`
   - Additionally query interfaces for this zone: `SELECT i.name FROM interfaces i WHERE i.pvid = n.vlan_id` (use the vlan_id from the first query)
   - Return JSON: `{ id, name, zone, vlan_id, subnet, gateway, dhcp_enabled, enabled, interfaces: [list of interface names] }`
   - 404 if zone not found

5. **Register routes** in the `protected_routes` router:
   ```rust
   .route("/api/v1/ports/{name}", get(port_get_handler).put(port_update_handler))
   .route("/api/v1/zones", get(zones_list_handler))
   .route("/api/v1/zones/{zone}", get(zone_get_handler))
   ```

**Important constraints:**
- All SQL uses parameterized queries (no string interpolation per CLAUDE.md)
- Error responses use `json!({ "error": "..." })` with appropriate status codes (404, 422)
- Use `tracing::info!` for successful operations, `tracing::warn!` for reconfig failures
- The port PUT only accepts `pvid` and `tagged_vlans` — no other fields (mtu, enabled use the existing interface_update endpoint)
- Port name validation: reject names with path traversal characters (`.`, `/`, `\`) before using in queries — defense in depth even though parameterized
