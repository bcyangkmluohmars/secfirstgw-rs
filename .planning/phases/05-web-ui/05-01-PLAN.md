Rewrite the switch panel rendering in Interfaces.tsx to use the PVID/tagged-VLAN model instead of the old `role` field. The page must fetch zone data and port data alongside the existing interface/board data.

**Data fetching changes:**
- In the `load()` callback, also call `api.getZones()` to get zones with their vlan_ids. Store in state as `zones: ZoneInfo[]`.
- Build a lookup map: `vlanToZone: Map<number, ZoneInfo>` (maps vlan_id -> zone info) from the zones array.
- For each port shown in the switch panel, call the existing interfaces data (which already has pvid/tagged_vlans from the GET /api/v1/interfaces endpoint). Check if `NetworkInterface` already has `pvid` and `tagged_vlans` fields — if NOT, also fetch per-port data via `api.getPort()` for each physical port.

IMPORTANT: Check what `GET /api/v1/interfaces` actually returns. If it already returns `pvid` and `tagged_vlans` per interface, use that data directly (no need for per-port API calls). If it only returns `role`, then fetch port data separately. Looking at the existing code, the `NetworkInterface` type has `role` but NOT `pvid`/`tagged_vlans` — so either:
  (a) Update the NetworkInterface type to include `pvid: number` and `tagged_vlans: number[]` and check the server returns them, OR
  (b) Fetch port data via getPort() for each physical port on load.

Prefer option (a) if the /api/v1/interfaces endpoint already returns pvid/tagged_vlans (check the Rust handler). If it doesn't, use option (b).

**Visual changes to renderPort() in both renderDeviceSwitch and renderGenericSwitch:**

1. **Primary color = PVID zone color:** Look up the port's `pvid` in the `vlanToZone` map to get the zone name, then use `zoneColor(zoneName)` for the port's primary styling (background, border, text).

2. **Void port (pvid=1) appearance:** If pvid === 1, render the port with a dark, inactive appearance regardless of link state:
   - Background: `bg-navy-950 border-navy-800/20`
   - Text: `text-navy-600`
   - No glow, reduced opacity (but NOT the same as "link down" — it should look intentionally disabled, not broken)
   - Show a small "VOID" label in 8px text below the connector type

3. **Tagged VLAN dots:** Below the port label (or in a row at the bottom of the port button), render small colored dots (w-1.5 h-1.5 rounded-full) for each VLAN in `tagged_vlans`. Each dot gets the color of the zone that owns that VLAN (via vlanToZone lookup). Skip the PVID's own VLAN — it's already represented by the primary color. If there are more than 3 tagged VLANs, show 3 dots + a "+N" counter in 7px text.

4. **WAN ports (pvid=0):** WAN ports have pvid=0 which signals "not an internal VLAN port". Keep existing WAN styling (red zone color). No tagged VLAN dots for WAN ports.

**Zone grouping changes:**
- Replace the `iface.role`-based zone grouping with PVID-based zone resolution. Group interfaces by looking up their pvid in vlanToZone to determine zone membership.
- The `ROLES` constant and `roleVariant` function can remain for backward compatibility with the zone cards section, but the switch panel MUST use PVID-based lookup.

**Remove old role-only patterns in the switch panel section:**
- `getPortZone()` should resolve zone from pvid via vlanToZone, NOT from `iface.role`.
- If an interface has no pvid data yet (e.g., virtual interfaces), fall back to the role field.

**Keep existing layout structure:**
- renderDeviceSwitch still shows LAN | MGMT | divider | WAN | SFP+ groups
- renderGenericSwitch still shows a grid of ports
- The board detection logic (`board ? renderDeviceSwitch(board) : renderGenericSwitch()`) stays the same
- The "Create VLAN" button and modal stay the same
- The zone cards section below the switch panel stays the same (can still use role for grouping there)
